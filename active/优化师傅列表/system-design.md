# 系统设计

> 编辑本文件前需遵守 `docs/standards/系统设计规范.md`，并按融合系统接口、数据库、环境和发布规范设计。

## 1. 背景与目标

- 背景：用户提出“优化师傅列表”，当前尚未提供具体产品描述、截图、入口或接口范围。
- 目标：基于既有师傅列表能力做最小必要优化，优先复用现有页面、接口、权限和数据模型。
- 非目标：不在本需求中默认重构师傅域模型、订单分配、结算、消息、MQ/Job 或新增独立服务。

## 2. 现状复核

涉及已有业务、已有接口、已有表、已有状态流、已有 MQ/Job/outbox/补偿链路时，先完成既有业务复核。

| 复核项 | 当前实现 | 本次处理方式 |
| --- | --- | --- |
| 入口 | GitNexus 初步命中 `worker-center` 的 `Workers/MemberController::list`，前端页面、菜单和角色路径待确认 | 复用现有入口，按确认范围优化。 |
| 数据模型 | GitNexus 初步显示列表字段来自 `member`、`member_services_grade`、`tmall_members`、`member_label` 等，实际表结构待代码和数据库规范复核 | 优先只读查询优化；不默认改表。 |
| 幂等与补偿 | 列表查询暂不涉及 | 如新增写操作或导出任务，再补幂等与补偿设计。 |
| 通知或事件 | 暂未发现需求涉及 | 不默认新增事件或消息。 |
| 调用链 | 候选链路：`Workers/MemberController::list -> MemberLogic::getList -> MemberLogic::getListByWhereCondition -> MemberRepository::getMemberList` | 先复核实际代码与路由，再决定是否扩展接口。 |

## 3. 总体方案

- 涉及应用：待确认前端应用、候选后端 `worker-center`。
- 关键流程：
  1. 用户进入师傅列表页面。
  2. 页面按现有权限加载筛选项和列表数据。
  3. 用户执行查询、分页、排序或查看列表字段。
  4. 系统返回符合权限边界和筛选条件的数据。
- 设计取舍：当前先按“既有能力增强”处理，避免在需求未澄清前引入新表、新状态或新服务。

## 6. 状态、幂等、异常与补偿

- 状态枚举：待复核现有师傅状态字段；本需求不默认新增状态。
- 幂等键：列表查询不涉及写入幂等。
- 重复请求或重复回调：列表查询允许重复请求，需保证分页和筛选结果稳定。
- 下游失败：若列表依赖 RPC/Feign 或外部系统，需定义降级或错误提示；当前待复核。
- 事务边界：列表查询默认只读事务或无显式事务。
- 补偿或重放：当前不涉及。
- 业务异常与错误码：沿用现有查询接口错误语义；新增参数需补参数校验和错误提示。

## 8. 安全与权限

- 当前用户来源：待复核当前列表接口的登录态、用户 Header、网关注入或前端权限。
- 角色路径：待确认。
- 数据权限：不得因优化查询或字段展示绕开现有门店、区域、组织或角色数据权限。
- 敏感字段：手机号、身份证、银行卡、地址等字段如展示或导出，需按现有脱敏规则处理。
- 日志脱敏：后端日志不得打印敏感字段明文。
